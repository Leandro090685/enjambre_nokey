# Breaking Changes: Odoo 18.0 → 19.0

## Python API Changes

### 1. `create()` Always Receives a List (CRITICAL)

In Odoo 19.0, `create()` ALWAYS receives `vals_list` (a list of dicts). This is the
most impactful change — every `create()` override must be updated.

```python
# BEFORE (18.0)
@api.model
def create(self, vals):
    if not vals.get('name'):
        vals['name'] = self.env['ir.sequence'].next_by_code('my.model')
    return super().create(vals)

# AFTER (19.0)
@api.model_create_multi
def create(self, vals_list):
    for vals in vals_list:
        if not vals.get('name'):
            vals['name'] = self.env['ir.sequence'].next_by_code('my.model')
    return super().create(vals_list)
```

Key points:
- `@api.model_create_multi` ensures the ORM normalizes a single dict into a list
  when callers pass `self.create({...})`
- WITHOUT the decorator, a single dict is NOT normalized — iterating gives string keys
- When post-processing the result, use `res.mapped('field')` to support multi-record sets
- Remove `@api.model` from create overrides — `@api.model_create_multi` replaces it

**Detection:**
```bash
grep -rn "def create(self, vals)" {module_path} --include="*.py"
grep -rn "@api.model" {module_path} --include="*.py"
```

### 2. `_sql_constraints` → `models.Constraint`

The list-based constraint syntax is replaced by class attributes.

```python
# BEFORE (18.0)
_sql_constraints = [
    ('unique_name', 'unique(name)', 'El nombre debe ser unico.'),
    ('check_amount', 'CHECK(amount >= 0)', 'El monto debe ser positivo.'),
]

# AFTER (19.0)
_unique_name = models.Constraint(
    'unique(name)',
    'El nombre debe ser unico.',
)
_check_amount = models.Constraint(
    'CHECK(amount >= 0)',
    'El monto debe ser positivo.',
)
```

Rules:
- The attribute name must start with `_` and be descriptive
- Supports `unique(...)` and `CHECK(...)`
- Requires `models` imported from `odoo`

**Detection:**
```bash
grep -rn "_sql_constraints" {module_path} --include="*.py"
```

### 3. `registry` Import Path Changed

```python
# BEFORE (18.0)
from odoo import registry
new_cr = registry(self.env.cr.dbname).cursor()

# AFTER (19.0)
from odoo.modules.registry import Registry
new_cr = Registry(self.env.cr.dbname).cursor()
```

If `registry` was imported but not used, just remove it from the import line.

**Detection:**
```bash
grep -rn "from odoo import.*registry" {module_path} --include="*.py"
grep -rn "from odoo.registry" {module_path} --include="*.py"
```

### 4. Controller `type='json'` Deprecated

For webhooks receiving plain JSON (not JSON-RPC), use `type='http'`.

```python
# BEFORE (18.0)
@http.route('/webhook', methods=['POST'], type='json', auth='none')
def webhook(self):
    data = json.loads(request.httprequest.data.decode('utf-8'))
    return {'status': 'ok'}

# AFTER (19.0)
@http.route('/webhook', methods=['POST'], type='http', auth='none', csrf=False)
def webhook(self):
    data = request.get_json_data()
    return Response(status=200)
```

Key points:
- `type='json'` is an alias for `type='jsonrpc'` (JSON-RPC 2.0 with id, method, params)
- For plain JSON webhooks, use `type='http'` + `csrf=False`
- Use `request.get_json_data()` instead of manual parsing
- Return `Response(status=200)` explicitly (import from `odoo.http`)

**Detection:**
```bash
grep -rn "type='json'\|type=\"json\"" {module_path} --include="*.py"
```

### 5. `_description` Mandatory on Models with `_name`

Every model defining `_name` must have `_description`. Missing it generates a WARNING.

```python
# BEFORE (18.0) — worked without _description
class MyModel(models.Model):
    _name = 'my.model'

# AFTER (19.0) — _description required
class MyModel(models.Model):
    _name = 'my.model'
    _description = 'Mi Modelo'
```

**Detection:**
```bash
grep -rn "_name\s*=" {module_path} --include="*.py"
# Then manually check each for matching _description
```

### 6. Related Fields: Don't Redefine `selection` or `string`

Related fields inherit `selection` and `string` from the source field automatically.
Redefining them generates a WARNING.

```python
# BEFORE (18.0)
field_x = fields.Selection(
    [('a', 'A'), ('b', 'B')],
    'Label',
    related='other_id.field_x',
    readonly=False,
)

# AFTER (19.0)
field_x = fields.Selection(
    related='other_id.field_x',
    readonly=False,
)
```

**Detection:**
```bash
grep -B5 "related=" {module_path}/models/ --include="*.py" | grep -E "selection|string"
```

```python
# BEFORE (18.0)
vals = {'product_uom': product.uom_id.id}

# AFTER (19.0)
vals = {'product_uom_id': product.uom_id.id}
```

**Detection:**


### 8. `mail.template` API Changed

```python
# BEFORE (18.0)
template.generate_email(res_id, ['body_html', 'subject'])
# Returns: {'body': '<html>...', 'subject': '...'}

# AFTER (19.0)
template._generate_template([res_id], ['body_html', 'subject'])
# Returns: {res_id: {'body_html': '<html>...', 'subject': '...'}}
```

Note: method name changed, parameter is now a list, return key is `body_html` not `body`.

**Detection:**
```bash
grep -rn "generate_email\|generate_template" {module_path} --include="*.py"
```

## XML / View Changes

### 9. `res.groups` Restructured with `res.groups.privilege`

Groups now use a privilege record to define their category/grouping.

```xml
<!-- BEFORE (18.0) -->
<record id="group_user" model="res.groups">
    <field name="name">Usuario</field>
    <field name="category_id" ref="module_category_xxx"/>
</record>
<record id="group_manager" model="res.groups">
    <field name="name">Manager</field>
    <field name="users" eval="[(4, ref('base.user_root'))]"/>
    <field name="category_id" ref="module_category_xxx"/>
    <field name="implied_ids" eval="[(4, ref('group_user'))]"/>
</record>

<!-- AFTER (19.0) -->
<record model="res.groups.privilege" id="privilege_xxx">
    <field name="name">Mi Privilegio</field>
    <field name="category_id" ref="module_category_xxx"/>
</record>
<record id="group_user" model="res.groups">
    <field name="name">Usuario</field>
    <field name="privilege_id" ref="privilege_xxx"/>
</record>
<record id="group_manager" model="res.groups">
    <field name="name">Manager</field>
    <field name="user_ids" eval="[Command.link(ref('base.user_root'))]"/>
    <field name="privilege_id" ref="privilege_xxx"/>
    <field name="implied_ids" eval="[Command.link(ref('group_user'))]"/>
</record>
```

Changes:
- Create a `res.groups.privilege` record to group the groups
- `category_id` on `res.groups` → `privilege_id`
- `users` → `user_ids`
- `(4, ref(...))` → `Command.link(ref(...))`

**Detection:**
```bash
grep -rn 'model="res.groups"' {module_path} --include="*.xml"
grep -rn "category_id.*ref=" {module_path}/security/ --include="*.xml"
grep -rn '"users"' {module_path}/security/ --include="*.xml"
grep -rn "(4, ref(" {module_path} --include="*.xml"
```

### 13. Duplicate Field Labels Warning

Odoo 19.0 warns when two fields on the same model have identical `string` values.
Ensure each field has a unique, descriptive label.

### 14. XML-RPC: `fields` as Keyword Argument

```python
# BEFORE (18.0) — fields as positional arg
models.execute_kw(db, uid, pw, model, 'read', [ids, ['f1', 'f2']])

# AFTER (19.0) — fields must be keyword arg
models.execute_kw(db, uid, pw, model, 'read', [ids], {'fields': ['f1', 'f2']})
```

**Detection:**
```bash
grep -rn "execute_kw" {module_path} --include="*.py"
```

### 15. `hr.contract` Model REMOVED → `hr.version` (CRITICAL)

The entire `hr.contract` model and the `hr_contract` addon were removed in Odoo 19.
Replaced by `hr.version` (defined in core addon `hr`, file `hr_version.py`). Each
employee now owns multiple `hr.version` records across time (`date_version`). Payroll
fields that used to extend `hr.contract` now extend `hr.version` (see enterprise
`hr_payroll/models/hr_version.py`).

Implications for custom modules:

- `depends: ['hr_contract']` in manifest → replace with `'hr'` (hr.version is defined there)
- `_inherit = 'hr.contract'` → typically `_inherit = 'hr.version'` if the field varies
  over time (salary, hours, etc.) or `_inherit = 'hr.employee'` if it's employee-level
- `comodel_name='hr.contract'` → `'hr.version'` (with relevant date filtering via
  `date_version`) or `'hr.employee'`
- There is NO compatibility layer — the table `hr_contract` does not exist

```python
# BEFORE (18.0)
class HrContract(models.Model):
    _inherit = 'hr.contract'
    my_custom_field = fields.Char()

# AFTER (19.0) — if field varies per contract period
class HrVersion(models.Model):
    _inherit = 'hr.version'
    my_custom_field = fields.Char()

# OR (19.0) — if field is employee-level
class HrEmployee(models.Model):
    _inherit = 'hr.employee'
    my_custom_field = fields.Char()
```

**Detection:**
```bash
grep -rn "_inherit.*hr\.contract[^.]" {module_path} --include="*.py"
grep -rn "comodel_name.*hr\.contract[^.]" {module_path} --include="*.py"
grep -rn "'hr\.contract'\|\"hr\.contract\"" {module_path} --include="*.py" --include="*.xml"
grep -rn "'hr_contract'" {module_path}/__manifest__.py
```

### 16. `_table` on `AbstractModel` — Remove

`AbstractModel` classes never create database tables. Defining `_table` on them is
unnecessary and may generate warnings in v19.

```python
# BEFORE (18.0)
class ReportMyReport(models.AbstractModel):
    _name = 'report.my_module.report_name'
    _description = 'Mi Reporte'
    _table = 'report_name'  # unnecessary

# AFTER (19.0) — remove _table
class ReportMyReport(models.AbstractModel):
    _name = 'report.my_module.report_name'
    _description = 'Mi Reporte'
```

**Detection:**
```bash
grep -rn "_table\s*=" {module_path} --include="*.py"
# Then check if the class inherits from AbstractModel
```
